Data Extension & Contact Builder

MCE 구축의 데이터 설계 핵심 — Data Extension 설계부터 Contact Builder 구성, Attribute Group 연결까지 실무 기준으로 정리합니다.

Data Extension 개요

Data Extension(DE)은 MCE의 데이터 저장 단위입니다. 관계형 데이터베이스의 테이블과 유사하며, 발송 대상 리스트·구매 이력·행동 데이터 등을 저장합니다. MCE는 List 방식도 지원하지만 DE 방식이 표준입니다 — 필드 커스터마이징, SQL 쿼리, Journey Builder 연동 등 모든 고급 기능이 DE에서만 동작합니다.

📌
List vs Data Extension: List는 단순 이메일 주소 모음으로 기능 제한이 큽니다. 신규 구축 시 반드시 DE 방식을 사용하세요. List는 하위 호환성을 위해 남아 있는 레거시 기능입니다.

4가지 DE 타입

타입설명데이터 입력 방식주요 용도
Standard 일반 목적 DE. 가장 많이 사용 수동 입력, Import, API Upsert, Query Activity Master DE, 발송 대상, 이벤트 데이터
Filtered 부모 DE의 서브셋. 필터 조건으로 자동 갱신 부모 DE에서 필터 자동 추출 (Refresh 필요) VIP 세그먼트, 특정 지역 대상, 등급별 추출
Random Filtered DE의 랜덤 샘플링 버전 부모 DE에서 지정 비율/건수 랜덤 추출 A/B 테스트 그룹 분리, 파일럿 발송
Synchronized SF CRM 오브젝트와 실시간 동기화 Marketing Cloud Connect 자동 동기화 CRM Contact/Lead 데이터 MCE에서 활용
⚠️
Filtered / Random DE 주의: 부모 DE가 변경되어도 자동 반영되지 않습니다. Automation Studio의 Refresh Group Activity로 주기적으로 갱신해야 최신 데이터가 반영됩니다.

필드 타입 레퍼런스

DE 생성 시 각 컬럼에 적합한 타입을 지정해야 합니다. 잘못된 타입 지정은 Import 오류, AMPscript 연산 오류의 주원인입니다.

타입길이 제한특이사항사용 예시
Text최대 4,000자가장 범용적. 숫자도 문자로 저장 가능이름, 주소, 코드값, 등급
Number-2³¹ ~ 2³¹소수점 없음. 정수만 저장나이, 포인트 수량, 순번
Decimal전체 18자리, 소수 10자리금액·비율 등 소수점 필요 시 사용구매금액, 할인율, 좌표
Date-저장 형식: M/D/YYYY 또는 YYYY-MM-DD. 시간 포함 시 DateTime 사용가입일, 구매일, 만료일
Boolean-True/False 또는 1/0. Import 시 "true"/"false" 문자열도 인식수신동의, 활성여부, 인증완료
EmailAddress254자이메일 형식 자동 검증. Sendable DE의 필수 필드 타입이메일 주소
Phone-MobileConnect SMS 발송 시 사용휴대폰 번호
Locale-언어-지역 코드 (예: ko-KR, en-US)다국어 발송 시 언어 분기
🚫
타입 변경 불가: DE 생성 후 필드 타입은 변경할 수 없습니다. 변경이 필요한 경우 DE를 새로 생성 후 데이터를 마이그레이션해야 합니다. 설계 단계에서 신중하게 결정하세요.

실무 필드 설계 팁

  • 날짜는 Date 타입으로: Text로 저장하면 SQL에서 날짜 연산(DATEADD, DATEDIFF) 불가
  • 금액은 Decimal로: Number는 소수점을 버려 실제 금액과 오차 발생
  • 전화번호는 Text로: 국가코드(+82), 하이픈(-) 포함 시 Phone 타입보다 Text가 안전
  • Primary Key는 Nullable 비허용: PK 필드에 NULL이 들어오면 Upsert 시 신규 레코드 생성 오류

시스템 Data Extension (Tracking DE)

MCE는 발송·오픈·클릭·바운스 등 모든 이벤트를 시스템 DE에 자동 기록합니다. 이름 앞에 밑줄(_)이 붙으며, Query Activity에서 직접 조회할 수 있습니다. 수동 수정은 불가합니다.

📌
보존 기간: 시스템 DE 데이터는 기본 6개월 보존 후 자동 삭제됩니다. 장기 분석이 필요한 경우 Automation Studio로 별도 DE에 주기적으로 복사해 두세요.
시스템 DE주요 필드설명
_Subscribers SubscriberKey, EmailAddress, Status, DateUnsubscribed AllSubscribers — MCE 전체 구독자 목록. Status: Active / Unsubscribed / Bounced / Held
_Sent SubscriberKey, JobID, ListID, BatchID, EventDate, TriggererSendDefinitionObjectID 발송 이벤트. 각 발송 건별 1레코드
_Open SubscriberKey, JobID, EventDate, IsUnique 오픈 이벤트. IsUnique=1이 최초 오픈
_Click SubscriberKey, JobID, EventDate, URL, IsUnique 클릭 이벤트. URL별로 각각 기록
_Bounce SubscriberKey, JobID, EventDate, BounceCategory, BounceSubcategory, SMTPCode 바운스 이벤트. BounceCategory: Hard / Soft / Block / Technical
_Unsubscribe SubscriberKey, JobID, EventDate, IsUnique 수신거부 이벤트 (Subscription Center 통한 거부)
_Job JobID, EmailName, EmailSubject, DeliveredTime, FromName, FromEmail 발송 작업 메타데이터. JobID로 각 시스템 DE를 JOIN
_SMSSubscriptionLog SubscriberKey, MobileNumber, OptInStatus, OptInDate SMS 구독 현황 (MobileConnect)
_PushSubscription SubscriberKey, DeviceID, PlatformType, OptInStatus Push 구독 현황 (MobilePush)

JobID로 발송 성과 집계 예시

SQL — 발송별 오픈율·클릭율 집계
SELECT j.JobID, j.EmailName, j.EmailSubject, j.DeliveredTime, COUNT(DISTINCT s.SubscriberKey) AS Sent, COUNT(DISTINCT o.SubscriberKey) AS Opens, COUNT(DISTINCT c.SubscriberKey) AS Clicks, COUNT(DISTINCT u.SubscriberKey) AS Unsubscribes, COUNT(DISTINCT b.SubscriberKey) AS Bounces, ROUND(COUNT(DISTINCT o.SubscriberKey) * 100.0 / NULLIF(COUNT(DISTINCT s.SubscriberKey),0),2) AS OpenRate, ROUND(COUNT(DISTINCT c.SubscriberKey) * 100.0 / NULLIF(COUNT(DISTINCT s.SubscriberKey),0),2) AS ClickRate FROM _Job j LEFT JOIN _Sent s ON j.JobID = s.JobID LEFT JOIN _Open o ON j.JobID = o.JobID AND o.IsUnique = 1 LEFT JOIN _Click c ON j.JobID = c.JobID AND c.IsUnique = 1 LEFT JOIN _Unsubscribe u ON j.JobID = u.JobID AND u.IsUnique = 1 LEFT JOIN _Bounce b ON j.JobID = b.JobID AND b.BounceCategory = 'Hard' WHERE j.DeliveredTime >= DATEADD(DAY, -30, GETDATE()) GROUP BY j.JobID, j.EmailName, j.EmailSubject, j.DeliveredTime ORDER BY j.DeliveredTime DESC

Sendable DE 설정

이메일을 발송하려면 DE를 Sendable로 지정해야 합니다. Sendable DE는 이메일 주소 필드와 Subscriber Key 필드를 MCE에게 명시적으로 알려주는 설정입니다.

1
DE 생성 또는 편집 화면 진입 → Used For Sending 체크박스 활성화
2
Send Relationship 설정: "EmailAddress 필드" relates to "Subscribers on" → Email Address 선택
3
Subscriber Key 매핑: DE의 Contact Key 역할 필드(예: MemberID)를 Subscriber Key에 연결
4
저장 후 Email Studio → Send Flow에서 해당 DE를 대상으로 선택 가능한지 확인
⚠️
Subscriber Key 일관성: Sendable DE에서 Subscriber Key로 매핑한 필드의 값은 AllSubscribers의 SubscriberKey와 반드시 일치해야 합니다. 불일치 시 Journey Builder에서 수신거부·바운스 처리가 제대로 작동하지 않습니다.

Shared Data Extension

Parent BU에서 생성한 DE를 Child BU에서 공유해 사용하는 기능입니다. 공통 회원 데이터나 글로벌 수신거부 목록을 여러 BU에서 함께 참조할 때 유용합니다.

항목설명
생성 위치반드시 Parent BU에서 생성 → DE 속성에서 Is Shareable 체크
접근 방법Child BU에서 Shared Data Extensions 폴더로 접근 (읽기 전용)
데이터 수정Parent BU에서만 가능. Child BU는 Read Only
Journey 활용Child BU의 Journey Builder에서 Shared DE를 Entry Source로 사용 가능
Query 활용Child BU의 Query Activity에서 [Enterprise].[데이터뷰명] 접두사로 조회
💡
활용 예시: Parent BU에 Global_Blacklist DE를 Shared로 생성 → 모든 Child BU의 Query Activity에서 발송 제외 목록으로 참조. BU별로 블랙리스트를 따로 관리하지 않아도 됩니다.

DE 명명 규칙

DE 이름은 추후 SQL 쿼리에서 직접 참조합니다. 일관된 명명 규칙이 없으면 운영 중 혼선이 발생합니다.

구분권장 패턴예시
Master DE[브랜드]_MasterBrandA_Master
발송 대상 추출 DE[브랜드]_Send_[캠페인코드]BrandA_Send_NL2024Q1
이력 DE[브랜드]_History_[유형]BrandA_History_Purchase
Journey 중간 저장 DE[브랜드]_JB_[Journey명]BrandA_JB_Welcome
Suppression(발송 제외)[브랜드]_Suppression_[사유]BrandA_Suppression_Blacklist
Tracking 복사본[브랜드]_Track_[이벤트]_[연도]BrandA_Track_Open_2024
  • 공백 대신 밑줄(_) 사용 — SQL에서 공백 있는 DE명은 대괄호로 감싸야 해 번거로움
  • 한글 이름 지양 — 시스템 간 인코딩 문제 가능성
  • 대소문자 구분: MCE SQL은 대소문자 무관하나 팀 내 표기 통일 권장

Contact Builder 개요

Contact Builder는 MCE의 데이터 관계 설정 도구입니다. 여러 DE를 Contact Key를 기준으로 연결하여 Attribute Group을 구성합니다. Journey Builder가 이메일 발송 시 수신자의 개인화 속성을 조회하는 소스가 됩니다.

📌
Contact Builder가 필요한 이유: Journey Builder에서 조건 분기("등급이 Gold인 경우") 또는 AMPscript Lookup 없이 DE 속성을 직접 참조하려면, 해당 DE가 Contact Builder의 Attribute Group에 연결되어 있어야 합니다.

All Contacts 구조

MCE의 연락처 계층 구조는 아래와 같습니다.

계층구성 요소설명
최상위All ContactsMCE 계정 전체의 모든 연락처 Pool. ContactKey 기준으로 자동 집계
2단계PopulationAttribute Group의 루트 DE. 해당 그룹의 "주 연락처 테이블" 역할
3단계Attribute Group관련 DE들을 묶은 논리 그룹. Journey Builder에서 이 단위로 데이터 참조
4단계연결된 DEContactKey로 Population DE에 조인된 추가 Data Extension
💡
Attribute Group 설계 팁: 브랜드(또는 BU)별로 하나의 Attribute Group을 기본으로 만들고, 필요에 따라 기능별(멤버십, 구매이력, 행동 데이터) 그룹을 추가하는 것이 유지보수에 유리합니다.

Attribute Group & Population 구성

Attribute Group 생성

1
Contact Builder 앱 접속 → Data Designer 탭 클릭
2
Create Attribute Group 클릭 → 이름 입력 (예: BrandA_ContactData)
3
생성된 그룹 위로 Master DE를 드래그 & 드롭 → Population으로 지정 (루트 DE)
4
Population DE에서 Contact Key 역할을 하는 필드를 Link로 지정

Population 설정 주의사항

  • Population으로 지정한 DE는 해당 Attribute Group의 루트(Root)가 됩니다
  • Population DE에 없는 ContactKey는 Journey Builder에서 해당 그룹의 데이터를 조회할 수 없습니다
  • Population DE는 Sendable DE와 동일한 DE를 사용하는 것이 일반적입니다 (Master DE)
  • 한 Attribute Group에 Population은 하나만 지정 가능합니다

Data Designer — DE 관계 연결

Data Designer에서 Population DE에 추가 DE를 연결하면 Journey Builder가 두 DE를 JOIN하여 속성을 가져올 수 있습니다.

관계 유형Cardinality의미예시
One-to-One 1:1 Contact 1명당 연결 DE에 레코드 1개 회원등급 DE, 최근구매일 DE, 포인트 DE
One-to-Many 1:N Contact 1명당 연결 DE에 레코드 여러 개 구매이력 DE, 방문이력 DE, 쿠폰 DE

DE 연결 절차 (Data Designer)

1
Attribute Group 화면에서 Link to Data Extension 클릭
2
연결할 DE 선택 → Cardinality 지정 (One-to-One 또는 One-to-Many)
3
조인 조건 설정: Population DE의 ContactKey 필드 ↔ 추가 DE의 ContactKey 필드 매핑
4
저장 후 Journey Builder의 Decision Split에서 해당 Attribute Group 속성 조회 테스트
⚠️
One-to-Many 주의: Journey Builder의 Decision Split에서 One-to-Many DE 속성을 조건으로 쓸 때는 어떤 레코드 기준으로 평가할지(최신 레코드, 첫 번째 레코드 등) 주의해야 합니다. 의도하지 않은 레코드가 기준이 될 수 있습니다. 가능하면 One-to-One DE를 준비해서 쓰는 것이 안전합니다.

Contact Key vs Subscriber Key

MCE에서 가장 혼동하기 쉬운 개념입니다. 두 키가 불일치하면 Journey Builder의 수신거부·바운스 처리가 깨집니다.

구분Contact KeySubscriber Key
사용 위치Contact Builder, Journey BuilderEmail Studio, AllSubscribers, Tracking DE
관리 주체Contact Builder / All ContactsAllSubscribers 테이블
역할데이터 관계(Attribute Group) 기준 키이메일 발송·추적 기준 키
기본값MCE가 자동 생성(GUID) 또는 직접 지정Sendable DE 설정 시 매핑한 필드값
🚫
반드시 동일한 값 사용: Contact Key와 Subscriber Key는 반드시 같은 값(예: 회원 ID)으로 설정해야 합니다. 구축 시 ContactKey 필드를 Sendable DE의 Subscriber Key로 매핑하면 자동으로 일치합니다. 이미 불일치 상태라면 데이터 마이그레이션이 필요합니다.

권장 키 설계

  • 신규 구축: 회원 ID(숫자 또는 UUID)를 Contact Key = Subscriber Key로 통일
  • 이메일 주소를 키로 쓰는 경우: 이메일 변경 시 키가 바뀌어 이력이 단절됨 — 가능하면 불변 ID 사용 권장
  • 기존 List 방식에서 DE 마이그레이션: 기존 List의 SubscriberKey를 ContactKey로 동일하게 맞추어 마이그레이션

핵심 SQL 패턴

Automation Studio의 Query Activity에서 사용하는 실무 SQL 패턴입니다. MCE SQL은 SQL Server 문법을 따릅니다.

발송 대상 추출 (수신거부·바운스 제외)

SQL — 발송 대상 추출
SELECT m.ContactKey, m.EmailAddress, m.Name, m.MemberGrade FROM BrandA_Master m -- AllSubscribers 수신거부자 제외 LEFT JOIN _Subscribers s ON m.ContactKey = s.SubscriberKey WHERE m.EmailOptIn = 1 AND m.IsActive = 1 AND m.EmailAddress IS NOT NULL AND (s.Status IS NULL OR s.Status = 'Active')

세그먼트 추출 — 등급별

SQL — VIP 등급 발송 대상
SELECT m.ContactKey, m.EmailAddress, m.Name, m.MemberGrade FROM BrandA_Master m LEFT JOIN _Subscribers s ON m.ContactKey = s.SubscriberKey WHERE m.EmailOptIn = 1 AND m.IsActive = 1 AND m.MemberGrade IN ('VIP', 'GOLD') AND (s.Status IS NULL OR s.Status = 'Active')

DE Upsert — Import 후 마스터 업데이트

SQL — Staging DE → Master DE Upsert
-- Staging DE(당일 CRM 추출본)를 Master DE에 반영 -- Query Activity의 Action을 "Update, Add" (Upsert)로 설정 SELECT s.ContactKey, s.EmailAddress, s.Name, s.EmailOptIn, s.IsActive, s.MemberGrade, s.LastPurchaseDate, GETDATE() AS UpdatedDate FROM BrandA_Staging s

시스템 DE 활용 쿼리

최근 30일 미오픈 구독자 추출

SQL — 휴면 세그먼트 (미오픈 30일)
SELECT DISTINCT m.ContactKey, m.EmailAddress, m.Name FROM BrandA_Master m LEFT JOIN _Subscribers s ON m.ContactKey = s.SubscriberKey WHERE m.EmailOptIn = 1 AND m.IsActive = 1 AND (s.Status IS NULL OR s.Status = 'Active') AND m.ContactKey NOT IN ( SELECT DISTINCT SubscriberKey FROM _Open WHERE EventDate >= DATEADD(DAY, -30, GETDATE()) )

Hard Bounce 주소 추출 → 비활성화

SQL — Hard Bounce 추출 (비활성화 대상)
-- Query Action: Update (IsActive를 False로 갱신) SELECT m.ContactKey, m.EmailAddress, m.Name, m.EmailOptIn, 0 AS IsActive, -- False로 업데이트 m.MemberGrade, m.LastPurchaseDate, GETDATE() AS UpdatedDate FROM BrandA_Master m INNER JOIN ( SELECT DISTINCT SubscriberKey FROM _Bounce WHERE BounceCategory = 'Hard' AND BounceDate >= DATEADD(DAY, -7, GETDATE()) ) b ON m.ContactKey = b.SubscriberKey WHERE m.IsActive = 1

발송 이력 장기 보관 (Tracking 복사)

SQL — _Open 데이터 연간 보관 DE로 복사
-- Query Action: Add (덮어쓰지 않고 누적) -- 대상 DE: BrandA_Track_Open_2024 SELECT o.SubscriberKey, o.JobID, j.EmailName, o.EventDate, o.IsUnique FROM _Open o INNER JOIN _Job j ON o.JobID = j.JobID WHERE o.EventDate >= '2024-01-01' AND o.EventDate < '2025-01-01'
