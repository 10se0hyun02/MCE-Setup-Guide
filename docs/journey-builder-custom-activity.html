R&R 정의 — MCE 구축 담당 vs. 슈어엠 개발사

Custom Activity 구현은 MCE 플랫폼 영역(본인)과 발송 모듈 개발 영역(슈어엠)이 명확히 나뉩니다. 아래 테이블은 도메인별 소유자, 산출물, 의사결정 권한을 정리한 것입니다.

도메인 소유자 산출물 / 책임 의사결정 권한
SFMC 앱 등록
Installed Packages
 MCE 담당 Custom Activity 컴포넌트 생성, Endpoint URL 입력, JWT Signing Secret 발급 SFMC 계정 접근 권한 보유자만 가능 — 슈어엠 개발자에게 위임 불가
중계 서버 구축
HTTPS + Endpoints
 슈어엠 /execute · /save · /publish 등 endpoint 개발, 서버 배포, HTTPS 인증서 적용 서버 언어·프레임워크·인프라 선택은 슈어엠 자율. URL 구조만 MCE와 사전 합의
config.json 작성
Activity 메타 정의
 슈어엠 config.json 파일 작성 및 서버 배포 (GET /config.json 응답) inArguments 필드명·타입은 MCE 담당자가 확정해서 전달, 나머지 구조는 슈어엠 작성
inArguments 설계
DE 컬럼 → API 매핑
 양측 협의 MCE DE 컬럼명 목록(MCE 담당 제공) ↔ 슈어엠 API 파라미터명(슈어엠 제공) 1:1 매핑 합의 최종 필드명은 양측 서면(메일·문서) 합의 후 config.json에 반영
DE 설계 및 연결
Contact Builder
 MCE 담당 발송 대상 DE 컬럼 확정, Attribute Group 연결, 휴대폰번호 형식 통일 데이터 구조·컬럼명 결정 권한은 MCE 담당. 형식 문제(하이픈 등) 전처리 방법도 MCE 결정 후 슈어엠에 통보
JWT 보안 처리
Signing Secret 관리
 양측 협의 MCE 담당: Secret 발급·전달 / 슈어엠: 환경변수 저장·검증 로직 구현 Secret 재발급은 MCE 담당 권한. 유출 시 즉시 MCE 담당에게 통보 후 재발급
슈어엠 API 연동
카카오 알림톡 / SMS
 슈어엠 /execute에서 슈어엠 내부 API 호출, 카카오 템플릿 등록·승인, 발송 결과 처리 슈어엠 내부 API 구조·호출 방식은 슈어엠 전담. MCE 담당은 결과(성공/실패 응답) 형식만 합의
발송 시나리오 설계
Journey 캔버스
 MCE 담당 Journey 설계, Custom Activity 노드 배치, 설정 팝업에서 템플릿 코드·발송 유형 입력 어떤 시나리오에서 어떤 템플릿을 쓸지는 MCE 담당 결정. 슈어엠은 템플릿 코드 목록 제공
통합 테스트
E2E 검증
 양측 협의 슈어엠: /execute 단독 테스트(Postman) 후 결과 공유 → MCE 담당: Journey Test Mode 실행 → 실 수신 확인 테스트 실패 시 원인 구간 판단: JWT 오류·inArguments 누락은 MCE 설정 문제 / 슈어엠 API 오류는 슈어엠 문제
발송 실패 처리 정책
오류 대응 설계
 양측 협의 슈어엠 오류 시 /execute가 200 반환해 Journey 계속할지, 오류 경로로 분기할지 사전 결정 비즈니스 정책 결정은 MCE 담당(또는 고객사). 결정 후 슈어엠이 구현
운영 모니터링
발송 후 확인
 영역 분리 MCE 담당: Journey Analytics에서 Activity 성공/실패 건수 확인 / 슈어엠: 발송 시스템 로그·콘솔에서 실 발송 결과 확인 불일치 발생 시(MCE 성공 / 슈어엠 미발송) 슈어엠 서버 로그 공유 요청

핵심 인터페이스 — 서로 주고받아야 할 정보

MCE → 슈어엠
① JWT Signing Secret (Installed Package 발급)  ·  ② inArguments 필드 목록 (필드명, 데이터타입, 필수여부)  ·  ③ 카카오 템플릿 코드 목록 (시나리오별 어떤 템플릿 쓸지)  ·  ④ 발송 실패 처리 정책 (200 반환 vs. 오류 분기)
슈어엠 → MCE
① 중계 서버 Endpoint Base URL (HTTPS)  ·  ② /execute Postman 테스트 결과 (성공 응답 예시)  ·  ③ 슈어엠 API 파라미터명 목록 (inArguments 매핑용)  ·  ④ 카카오 템플릿 코드 목록 (슈어엠 콘솔 기준)  ·  ⑤ 등록된 발신번호 목록 (SMS/LMS 발송용 사전등록 완료 번호)
💡
R&R 경계선 한 줄 요약: MCE 담당은 "무엇을 누구에게 언제 발송할지"를 결정하고, 슈어엠은 "받은 데이터로 실제 메시지를 어떻게 전송할지"를 구현합니다. 두 역할이 만나는 지점이 /execute endpoint입니다.

역할별 구현 체크리스트

MCE 담당 항목은 본인 영역, 슈어엠 개발자 항목은 상대방에게 요청·확인해야 할 작업입니다. 노란색 항목은 양측 협의가 필요합니다.

🔵 MCE 담당자 (본인)
🟢 슈어엠 개발자 담당

⚡ 양측 협의 필요 — 착수 전 확정

① inArguments 항목 확정 — MCE DE 컬럼명과 슈어엠 API 파라미터명 1:1 매핑 합의 (예: Phone → recipientNo)
② 템플릿 코드 관리 방식 — Journey마다 다른 템플릿을 쓸 경우 config.html UI에서 입력할지, DE 컬럼으로 전달할지 결정
③ 발송 실패 처리 정책 — 슈어엠 오류 시 /execute가 200을 반환해 Journey를 계속 진행할지, 오류 경로로 분기할지 결정
④ 발송 결과 콜백 여부 — 슈어엠 결과를 MCE DE에 역적재할지, 슈어엠 콘솔에서만 확인할지 결정
⑤ Fail-Back(대체 발송) 정책 — 카카오 알림톡 실패 시 SMS/LMS 자동 재전송 여부 결정. 슈어엠 지원 여부 확인 후 inArguments에 SMS 내용 필드 추가 필요 여부 판단
⑥ 080 수신거부 연동 여부 — 광고성 메시지(친구톡 광고형, SMS 광고) 발송 시 해당. 슈어엠의 수신거부 처리 방식 확인 및 MCE 구축사 서버 수신 endpoint 구현 필요 여부 결정

개요

Custom Activity는 Journey Builder 캔버스에서 외부 REST API를 직접 호출할 수 있게 해주는 확장 기능입니다. MCE 기본 제공 채널(이메일, SMS, 푸시) 외에 사내 시스템·외부 플랫폼과 연동할 때 사용합니다.

세스코 프로젝트에서는 슈어엠(카카오 알림톡 / 문자) 발송 시스템을 Journey Builder에서 트리거하기 위해 Custom Activity를 구현합니다.

구분내용
용도Journey 진행 중 슈어엠 API 호출 → 카카오 알림톡 / LMS 발송
구현 방식MCE Custom Activity (REST) + 중계 서버 (Node.js / Spring 등)
MCE에서 호출하는 URL외부 중계 서버의 /execute endpoint
인증SFMC 발행 JWT (HS256) — 중계 서버에서 검증
발송 대상 데이터DE의 휴대폰번호, 고객명 등 → inArguments로 전달

동작 구조

Customer가 Journey에 진입하면 MCE가 Custom Activity endpoint(/execute)를 호출하고, 중계 서버가 이를 받아 슈어엠 API를 호출하는 2-hop 구조입니다.

단계주체동작
①MCEContact가 Custom Activity 노드에 도달하면 /execute에 JWT + inArguments POST
②중계 서버JWT 서명 검증 → inArguments에서 전화번호·템플릿 코드 추출
③중계 서버슈어엠 API 호출 (카카오 알림톡 or LMS)
④중계 서버{"status":"ok"} 200 응답 → MCE가 다음 Activity로 진행
⑤슈어엠최종 고객에게 메시지 전송 + 결과 콜백 (선택)
ℹ️
MCE는 응답을 기다립니다: /execute 기본 타임아웃은 20초(20000ms)이며, config.json의 execute.timeout으로 1000~100000ms 범위에서 조정 가능합니다. 타임아웃 초과 시 Activity 실패 처리됩니다.

사전 준비

패키지 앱 등록 (SFMC)

📋
등록 전 먼저 받아야 할 것: 슈어엠이 중계 서버를 구축하고 HTTPS Base URL을 확정한 뒤에 등록을 시작하세요. URL 없이는 진행이 안 됩니다.

2
New 클릭 → 패키지 이름 입력
예: SureM Custom Activity — SFMC 내부 관리용 이름, JB 캔버스에는 표시되지 않음
3
Add Component 클릭 → Journey Builder Activity 선택
4
아래 3개 필드 입력
필드입력값비고
Name JB 캔버스에 표시될 이름
(예: 슈어엠 알림톡 발송)		 고객사 담당자가 보는 이름
Key 직접 입력하거나 자동 생성된 값 확인
(예: surem-kakao-activity 또는 UUID)		 config.json의 applicationExtensionKey와 반드시 일치해야 함 — 저장 후 슈어엠에 전달
Endpoint URL 슈어엠이 전달한 서버 Base URL
(예: https://sure-m.com/mce-activity)		 SFMC가 뒤에 /config.json 붙여서 GET 호출
5
Save 클릭 → 패키지 상세 페이지에서 JWT Signing Secret 확인
Secret은 최초 발급 시에만 표시됩니다. 반드시 복사해두세요.
6
슈어엠에게 아래 두 값 전달
🔑 JWT Signing Secret — 슈어엠이 환경변수로 저장, /execute 요청 서명 검증에 사용
🔖 applicationExtensionKey — 슈어엠이 config.json configurationArguments.applicationExtensionKey 필드에 기재
⚠️
HTTPS 필수: MCE는 HTTP endpoint를 허용하지 않습니다. 슈어엠 서버에 유효한 SSL 인증서가 적용되어야 합니다. 개발 단계에서는 ngrok으로 로컬 서버를 임시 HTTPS 터널링해서 테스트할 수 있습니다.

config.json 작성

MCE는 앱 등록 후 {Endpoint URL}/config.json을 GET 요청해 Custom Activity의 메타정보를 읽어옵니다. 이 파일이 핵심 설정입니다.

전체 구조 (슈어엠 연동 예시)

{
  "workflowApiVersion": "1.1",

  // Journey Builder UI에 표시되는 정보
  "metaData": {
    "icon":      "images/icon.png",
    "iconSmall": "images/iconSmall.png",
    "category":  "message"   // 공식 유효값: message | customer | flow | custom
  },

  "type": "Rest",   // 공식값: Rest | RestDecision (대소문자 주의)

  "lang": {
    "en-US": {
      "name":        "슈어엠 발송",
      "description": "카카오 알림톡 / SMS 발송 (슈어엠 연동)"
    }
  },

  // execute 호출 시 MCE → 중계서버로 전달할 데이터 정의
  "arguments": {
    "execute": {
      "inArguments": [
        { "contactKey":   "{{Contact.Key}}" },
        { "phone":         "{{Attribute.세스코몰DE.Phone}}" },
        { "customerName":  "{{Attribute.세스코몰DE.Name}}" },
        { "templateCode":  "" },   // 설정 UI에서 입력
        { "messageType":   "" }    // "kakao" | "sms"
      ],
      "outArguments": [],
      "url":                "https://your-server.com/custom-activity/execute",
      "timeout":            20000,    // 기본 20000ms, 범위 1000~100000
      "retryCount":         0,        // 타임아웃 시 재시도 횟수 (0~5, 기본 0)
      "retryDelay":         1000,     // 재시도 대기 ms (0~10000, 기본 1000)
      "concurrentRequests": 6         // 병렬 실행 수 (1~10, 기본 6)
    }
  },

  // Journey lifecycle endpoint — 각 시점마다 MCE가 POST 호출
  "configurationArguments": {
    "applicationExtensionKey": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx", // 필수: Installed Package 키
    "save":      { "url": "https://your-server.com/custom-activity/save",      "useJwt": true },
    "publish":   { "url": "https://your-server.com/custom-activity/publish",   "useJwt": true },
    "validate":  { "url": "https://your-server.com/custom-activity/validate",  "useJwt": true },
    "unpublish": { "url": "https://your-server.com/custom-activity/unpublish", "useJwt": true },
    "stop":      { "url": "https://your-server.com/custom-activity/stop",      "useJwt": true }
  },

  // 설정 팝업 UI (템플릿 코드, 발송 유형 선택 등)
  "userInterfaces": {
    "configModal": {
      "url":        "https://your-server.com/custom-activity/config.html",
      "fullscreen": false,
      "height":     300
    }
  },

  // inArguments 타입 스키마 (MCE 유효성 검사용)
  "schema": {
    "arguments": {
      "execute": {
        "inArguments": [
          { "contactKey":   { "dataType": "Text", "isNullable": false, "direction": "in" }},
          { "phone":         { "dataType": "Text", "isNullable": false, "direction": "in" }},
          { "customerName":  { "dataType": "Text", "isNullable": true,  "direction": "in" }},
          { "templateCode":  { "dataType": "Text", "isNullable": false, "direction": "in" }},
          { "messageType":   { "dataType": "Text", "isNullable": false, "direction": "in" }}
        ],
        "outArguments": []
      }
    }
  }
}

execute 주요 파라미터 (공식 문서 기준)

속성기본값범위설명
timeout20000ms1000 ~ 100000응답 대기 시간. 초과 시 Activity 실패 처리
retryCount00 ~ 5타임아웃 시 재시도 횟수. 슈어엠 호출 중복 주의
retryDelay1000ms0 ~ 10000재시도 전 대기 시간
concurrentRequests61 ~ 10동시 병렬 실행 수. 슈어엠 API 부하 테스트 후 조정
⚠️
retryCount 주의: 재시도 시 /execute가 다시 호출되므로 슈어엠 발송이 중복될 수 있습니다. 슈어엠 API에 멱등성(idempotency) 처리가 없다면 retryCount: 0을 유지하세요.

inArguments 설계 포인트

필드명MCE 표현식설명
contactKey{{Contact.Key}}MCE 고객 식별자 (항상 포함 권장)
phone{{Attribute.DE명.컬럼명}}휴대폰번호 — DE의 컬럼명과 정확히 일치해야 함
templateCode(설정 UI에서 입력)슈어엠 발급 카카오 템플릿 코드

DE 설계 결정 기준

MCE는 DE 데이터를 그대로 inArguments로 넘기므로, 슈어엠 API 스펙이 최종 기준입니다. DE 설계 전에 슈어엠에서 API 파라미터 목록과 형식을 먼저 받아야 합니다.

1

2

백엔드 Endpoint 구현

/execute — 실제 발송 처리

POST /custom-activity/execute Contact가 Activity 노드 도달 시 MCE가 호출

MCE가 JWT Bearer 토큰 방식으로 요청을 보냅니다. JWT payload의 inArguments에서 발송 대상 데이터를 추출합니다.

Node.js 구현 예시

const express = require('express');
const jwt     = require('jsonwebtoken');
const axios   = require('axios');

const JWT_SECRET = process.env.SFMC_JWT_SECRET;   // Installed Package에서 복사한 값
const SUREM_API  = process.env.SUREM_API_URL;
const SUREM_KEY  = process.env.SUREM_API_KEY;

app.post('/custom-activity/execute', async (req, res) => {
  try {
    // 1) JWT 검증
    const token   = req.headers.authorization?.replace('Bearer ', '');
    const decoded = jwt.verify(token, JWT_SECRET);

    // 2) inArguments 추출 (배열 중 첫 번째 요소)
    const args = decoded.inArguments?.[0] ?? {};
    const { phone, customerName, templateCode, messageType } = args;

    if (!phone || !templateCode) {
      return res.status(400).json({ status: 'error', message: '필수 파라미터 누락' });
    }

    // 3) 슈어엠 API 호출
    if (messageType === 'kakao') {
      await axios.post(`${SUREM_API}/kakao/send`, {
        apiKey:       SUREM_KEY,
        recipientNo:  phone,
        templateCode: templateCode,
        templateParameter: { name: customerName }
      });
    } else {
      await axios.post(`${SUREM_API}/sms/send`, {
        apiKey:      SUREM_KEY,
        recipientNo: phone,
        content:     args.message
      });
    }

    // 4) 성공 응답 (MCE가 다음 Activity로 진행)
    res.status(200).json({ status: 'ok' });

  } catch (err) {
    console.error('execute error', err.message);
    res.status(500).json({ status: 'error' });
  }
});
🚫
500 응답 시 Journey 진행 중단: /execute가 4xx/5xx를 반환하면 MCE는 해당 Contact를 오류 경로(Error Exit)로 분기합니다. 슈어엠 발송 실패가 전체 Journey를 막지 않도록 발송 로직을 try-catch로 감싸고, 실패 시에도 200을 반환하는 정책을 검토하세요.

POST /custom-activity/save 마케터가 Journey를 저장할 때마다 호출

/publish — Journey 활성화 시 호출

POST /custom-activity/publish Journey Activate 시 1회 호출

발송 준비 완료 확인(슈어엠 템플릿 코드 유효성 사전 검증 등)에 활용할 수 있습니다. 200 이외 응답 시 Journey 활성화가 실패합니다.

app.post('/custom-activity/publish', async (req, res) => {
  // 선택: 슈어엠 템플릿 코드 유효성 사전 확인
  res.status(200).json({});
});
ℹ️
전체 lifecycle endpoint (공식 문서 기준): save · publish · validate · unpublish · stop 5종. 미구현 시 configurationArguments에서 해당 항목을 생략하면 됩니다. 최소 구현은 save + publish입니다.

080 수신거부 연동

광고성 메시지(친구톡 광고형, SMS 광고 등) 발송 시 정보통신망법상 수신거부 처리 의무가 있습니다. 슈어엠과 수신거부 연동 방식을 사전에 협의하세요.

⚠️
알림톡은 해당 없음: 카카오 알림톡은 거래 · 안내성 메시지로 분류되어 광고성 규제가 적용되지 않습니다. 친구톡 광고형이나 SMS 광고 발송 시에만 080 수신거부 연동이 필요합니다.
항목내용
일반적인 080 연동 방식 발송 서버가 고객사 서버로 수신거부 데이터를 PUSH → 고객사가 수신 endpoint 구현 후 수신거부 DB 적재
슈어엠 방식 슈어엠 포털 내 수신거부 관리 or 별도 API 제공 여부 — 슈어엠에 별도 확인 필요
수신거부 번호 형식 하이픈 없는 숫자만 (예: 01012345678)
방화벽 고려 슈어엠 서버 → MCE 구축사 서버 inbound 허용 필요 여부 확인
ℹ️
세스코 초기 구축: 알림톡(AT) 단일 채널이라면 080 수신거부 연동은 당장 불필요합니다. 향후 친구톡 광고형 또는 SMS 광고 도입 시 슈어엠과 별도 협의하세요.
