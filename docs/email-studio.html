Email Studio 개요

Email Studio는 Marketing Cloud Engagement(MCE)의 핵심 앱으로, 이메일 작성·발송·구독자 관리·성과 추적을 담당합니다. 발송 유형에 따라 세 가지 채널로 구분됩니다.

발송 유형설명주요 도구
Promotional 뉴스레터·캠페인 등 마케팅 목적 대량 발송. Commercial send classification 사용 Guided Send, User-initiated Send
Transactional 주문확인·비밀번호 재설정 등 트랜잭션 이메일. 수신거부된 구독자에게도 발송 가능 Triggered Send (Transactional classification)
Triggered API 호출·이벤트에 의해 개인별 자동 발송. Journey Builder와 통합 운용 Triggered Send Definition, REST API
ℹ️
Commercial vs Transactional: Send Classification 설정이 발송 유형을 결정합니다. Transactional으로 설정하면 수신거부(Unsubscribed) 상태의 구독자에게도 이메일이 전송되므로 반드시 실제 트랜잭션성 이메일에만 사용해야 합니다.

SAP — Sender Authentication Package

SAP는 MCE 계정의 이메일 발신 신뢰도를 높이는 인증 패키지입니다. 다음 세 가지 구성요소로 이루어집니다.

구성요소설명효과
Private Domain 발신 도메인을 @brand.com으로 설정 (기본 @salesforce.com 대신) 브랜드 도메인 기반 SPF/DKIM 인증, 도메인 레퓨테이션 직접 관리
Custom Branding (Tracking Domain) 링크 추적·이미지 호스팅 URL을 브랜드 도메인으로 변경 (click.brand.com) 피싱 경고 방지, URL 일관성, 브랜드 신뢰도 향상
Dedicated IP 공유 IP 대신 고객 전용 IP 할당 타 고객 발송 행동에 영향받지 않는 독립 레퓨테이션 확보, IP Warming 필요
⚠️
Dedicated IP 전환 시: IP Warming 없이 갑자기 대량 발송하면 스팸 필터에 차단될 수 있습니다. 최소 4~6주 워밍업 스케줄(하루 200건→500건→1,000건→…)을 계획하세요.

Send Logging

발송 런타임 정보(수신자, 발송일시, 발송 결과 등)를 Data Extension에 실시간 저장하는 기능입니다.

  • Email Studio → Admin → Send Logging에서 활성화
  • 로그를 저장할 DE를 지정하고 저장할 필드를 선택 (JobID, BatchID, SubscriberKey, EmailName, EventDate 등)
  • AMPScript로 %%[SET @jobid = AttributeValue("jobid")]%% 방식으로 런타임 값 참조 가능
  • Intelligence Reports와 연동해 발송 분석 확장에 활용
ℹ️
용도: 고객별 발송 이력 추적, A/B 테스트 결과 분석, 개인화 배치 발송 감사에 유용합니다. 데이터 보존 기간은 DE 설정에 따라 달라집니다.

Transactional Reconciliation

Triggered Send의 최종 발송 상태를 추적하는 기능입니다. 예상 발송 수 대비 실제 완료·실패 건수를 집계할 수 있습니다.

구성요소설명
Sendable_Reconcilable_Data_Extension 트랜잭션 발송 요청을 기록하는 DE 템플릿. 필수 필드: TransactionKey(중복 방지), EmailAddress, SubscriberKey, Status
_ReconcilableDispositionView 시스템 Data View. 각 트랜잭션의 최종 처리 상태(발송완료/에러/수신거부 등)를 조회할 수 있음 
-- 트랜잭션 발송 결과 집계 쿼리
SELECT
  rdv.TransactionKey,
  rdv.Status,        -- Sent / Error / Unsubscribed 등
  rdv.ErrorCode,
  rdv.EventDate
FROM _ReconcilableDispositionView rdv
JOIN Sendable_Reconcilable_Data_Extension src
  ON rdv.TransactionKey = src.TransactionKey
WHERE rdv.EventDate > DATEADD(day, -1, GETDATE())

Auto-Suppression List

특정 이메일 주소 또는 도메인을 모든 발송에서 자동으로 제외하는 기능입니다.

  • Email Studio → Admin → Auto-Suppression Lists에서 관리
  • 적용 범위: 선택한 BU 또는 전체 Account
  • 활용 사례: 내부 테스트 주소 보호, 경쟁사 도메인 제외, VIP 고객 발송 오류 방지
  • Exclusion List와 달리 Journey·Automation 발송 모두에 자동 적용됨

Import 요구사항

Lists 또는 Data Extension으로 구독자를 일괄 임포트할 때 파일 형식과 열 순서를 맞춰야 합니다.

항목요구사항
파일 형식CSV 또는 Tab-delimited (.txt)
헤더 행권장 (첫 번째 행이 열 이름). 없으면 열 순서로 매핑
이메일 열 위치마지막 열에 배치 권장 — 중간 열에 있으면 파싱 오류 가능
인코딩UTF-8 권장. BOM 없이 저장
최대 파일 크기단일 임포트 파일: 최대 10 MB (FTP 배치 임포트 시 제한 없음)
중복 처리기본: 최신 레코드로 업데이트. Lists는 SubscriberKey 기준 중복 제거
⚠️
이메일 열 위치: 이메일 주소를 반드시 파일의 마지막 열에 배치하세요. Salesforce 공식 문서의 명시 권고사항입니다.

구독자 상태 5종

모든 구독자는 이메일 주소별로 상태가 관리됩니다. 상태에 따라 이메일 수신 여부가 결정됩니다.

Active
정상 수신 상태. Commercial 및 Transactional 이메일 모두 수신
Bounced
Hard Bounce 1회 발생 시 전환. Commercial 발송 중단. Transactional은 계속 수신 가능
Held / Undeliverable
Soft Bounce 3회 누적(15일 이상 경과 조건) 시 전환. 모든 발송 차단
Unsubscribed
수신거부 처리됨. Commercial 발송 차단. Transactional classification은 발송 가능
Deleted
계정에서 완전 삭제. 재구독·재임포트 시 Active로 복원 불가(별도 처리 필요)
ℹ️
Held 전환 조건 (Soft Bounce): Soft Bounce 3회 발생 + 첫 바운스 이후 15일이 경과해야 Held로 전환됩니다. 15일 이내 3회 바운스는 카운트만 누적하고 상태가 유지됩니다.

Bounce 유형 분류

이메일 바운스는 원인에 따라 4가지로 구분됩니다. 유형별로 상태 전환 규칙이 다릅니다.

Soft Bounce
일시적 수신 불가
원인:
• Inactive Account (계정 비활성)
• Mailbox Full (사서함 용량 초과)
• Temporary Domain Failure (도메인 일시 장애)
• Other (기타 일시적 원인)

전환: 3회 누적 + 15일 경과 → Held
Hard Bounce
영구적 수신 불가
원인:
• User Unknown (존재하지 않는 주소)
• Domain Not Found (존재하지 않는 도메인)

전환: 1회 발생 즉시 → Bounced

재활성화: Email Studio → Subscribers에서 수동 Active 변경 후 재테스트 필요
Block Bounce
ISP 차단
원인:
• ISP가 발신 IP 또는 도메인을 스팸 차단 목록에 등재
• 스팸 신고율 과다로 인한 ISP 정책 차단

대응: Postmaster Tools(Gmail/Yahoo) 확인, Salesforce Support에 IP 평판 조회 요청, 발송 목록 정제
Technical Bounce
기술적 오류
원인:
• 수신 서버 구성 오류
• 연결 타임아웃
• 메시지 크기 초과

특징: Soft/Hard/Block으로 분류되지 않는 기타 기술적 실패. 구독자 상태에는 바로 영향 없음
✅
바운스 관리 팁: Automation Studio에서 _bounce Data View를 주기적으로 쿼리하여 Hard Bounce / Block Bounce 누적 주소를 정제하면 전체 도메인 레퓨테이션을 유지할 수 있습니다.

수신거부 관리 — 3단계

수신거부는 범위에 따라 세 단계로 처리됩니다.

Level 1
List-level Unsubscribe
특정 List 또는 Publication List에서만 수신거부. 다른 리스트의 이메일은 계속 수신. Publication Lists 세분화 시 권장
Level 2
Universal Unsubscribe
"Unsubscribe from All" — 계정 전체 Commercial 이메일 수신거부. _subscribers 테이블에 Unsubscribed 상태 기록
Level 3
Feedback Loop (FBL)
ISP(Gmail·Yahoo 등)가 스팸 신고를 Salesforce에 ARF 형식으로 전달. 자동으로 수신거부 처리됨

List-Unsubscribe Header

이메일 헤더에 수신거부 링크를 추가해 ISP 인터페이스(Gmail 상단 "Unsubscribe" 버튼)에 표시되도록 합니다.

방식형식특징
Mail-to List-Unsubscribe: <mailto:unsub@brand.com?subject=unsub> 이메일 기반 수신거부. 구형 ISP 지원
HTTP (One-Click) List-Unsubscribe: <https://click.brand.com/unsub?...> + List-Unsubscribe-Post: List-Unsubscribe=One-Click RFC 8058 One-Click. Gmail·Yahoo 2024 발송자 정책 필수 요건
🔴
2024 Gmail/Yahoo 발송자 정책: 하루 5,000건 이상 발송 시 One-Click Unsubscribe(RFC 8058) 구현이 필수입니다. MCE SAP Custom Tracking Domain 설정 후 Email Studio에서 자동 삽입 옵션을 활성화하세요.

페이지기능URL 변수

Lists vs Data Extensions

구독자 저장 방식을 Lists로 할지 Data Extension으로 할지는 프로젝트 규모와 개인화 요구에 따라 결정합니다.

기준ListsData Extensions
구독자 관리All Subscribers 연동. 구독 상태 자동 관리독립 테이블. 상태 관리는 _subscribers 별도 조회
필드 구성고정 Profile Attributes 사용완전 커스텀 필드 설계 가능
개인화기본 Attribute 참조만 가능복잡한 DE 관계·Lookup 함수 사용 가능
B2C 대규모수백만 건 이상 비권장대용량 최적화. 인덱스·파티션 설계 가능
Automation 연동제한적 (Query 결과를 List로 출력 불가)SQL Query → DE 출력 → Journey 진입 최적
권장 시나리오소규모·단순 이메일 발송, Lists 기반 기존 환경 유지신규 구축 시 항상 DE 방식 권장

Publication Lists

구독자가 수신거부할 수 있는 리스트의 세분화 단위입니다. "뉴스레터", "프로모션", "이벤트 알림" 등 유형별로 별도 Publication List를 만들어 관리합니다.

  • 장점: 구독자가 전체 수신거부 대신 원하는 유형만 해지 가능 → 이탈률 감소
  • Subscription Center 연동: %%subscription_center_url%%에서 각 Publication List의 구독·해지 UI 자동 생성
  • AMPScript 조회: LookupRows("_ListSubscribers","SubscriberKey",@sk)로 구독 여부 확인 가능

Triggered Sends

Triggered Send는 API 호출 또는 시스템 이벤트로 개별 구독자에게 즉시 이메일을 발송하는 기능입니다. 주문확인·비밀번호 재설정·환영 이메일 등 실시간 트랜잭션 발송에 최적입니다.

List Detective

Triggered Send 실행 시 자동으로 수신자 목록을 스캔하여 발송 레퓨테이션을 보호하는 기능입니다.

  • 스팸 트랩 주소 탐지 (ISP가 의도적으로 배포한 모니터링 주소)
  • 알려진 불량 이메일 주소(Role-based address: admin@, info@, noreply@ 등) 제외
  • 차단된 주소는 에러 코드 120으로 기록
  • Admin → List Detective에서 스캔 수준(Standard / Aggressive) 설정 가능

A/B Testing

두 가지 이메일 버전(Condition A / B)을 테스트 그룹에 발송한 후, 성과 기준에 따라 승리 버전을 나머지 구독자에게 자동 발송합니다.

승자 선정 기준

  • 최고 고유 오픈율(Highest Unique Open Rate): 테스트 기간 종료 시 Condition A/B 중 고유 오픈율이 높은 버전을 승자로 선택
  • 최고 클릭율(Highest Click-Through Rate): 클릭율이 높은 버전 선택
  • 승자 결정 후 나머지 구독자(Remainder)에게 자동 발송

Test Distribution

A/B 테스트 시작 시 시스템이 먼저 전체 수신자를 분할합니다. 대용량 리스트에서는 분할 처리에 시간이 소요될 수 있으므로 예약 발송 시 여유 시간을 확보하세요.

A/B 테스트 상태설명
Unscheduled생성되었으나 예약 미완료
Scheduled발송 예약 완료, 대기 중
InitializingA/B/Remainder 그룹 분할 처리 중
ActiveCondition A·B 발송 완료, 결과 수집 중
Complete승자 확정, Remainder 발송 완료
ℹ️
Dynamic Content와 A/B 테스트: Dynamic Content 또는 AMPScript가 포함된 이메일에도 A/B 테스트를 실행할 수 있습니다. 단, 두 개의 이메일 본문으로 테스트할 때 Dynamic Subject Line이 있으면 기본(default) 제목이 사용됩니다.

Send Throttling

시간당 발송 수를 제한하여 ISP에 과도한 트래픽이 집중되는 것을 방지합니다.

  • Email Studio → Admin → Send Throttling에서 설정
  • 단위: 시간당 발송 건수
  • 주요 활용 시나리오:
    • 신규 IP Warming 단계: 시간당 200건→500건→1,000건 점진적 증가
    • 대용량 캠페인 발송 시 ISP별 조절
    • 서버 부하 분산 (백엔드 처리 시스템 보호)

Multipart MIME

HTML 버전과 Plain Text 버전을 동시에 포함하는 이메일 형식입니다. 이메일 클라이언트가 지원 유형에 따라 자동으로 적합한 버전을 표시합니다.

설정 위치방법
Triggered SendTriggered Send Definition 설정 시 "Multipart MIME" 체크박스 활성화
Standard Send이메일 발송 마법사 → 고급 설정에서 Plain Text 버전 추가
✅
권장: Triggered Send(주문확인·알림 등)는 Multipart MIME을 항상 활성화하세요. 일부 기업용 이메일 클라이언트는 Plain Text만 처리하며, SPF/DKIM 검증에도 도움이 됩니다.

에러 코드 활용 쿼리

-- 최근 24시간 발송 에러 코드별 집계
SELECT
  sl.ErrorCode,
  COUNT(*) AS ErrorCount
FROM SendLog_DE sl          -- Send Logging DE 이름으로 변경
WHERE sl.EventDate > DATEADD(hour, -24, GETDATE())
  AND sl.ErrorCode IS NOT NULL
GROUP BY sl.ErrorCode
ORDER BY ErrorCount DESC
