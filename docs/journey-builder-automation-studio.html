Journey Builder & Automation Studio

Automation Studio로 데이터 처리 워크플로를 자동화하고, Journey Builder로 고객 여정을 설계합니다. 이 두 앱은 MCE 구축의 핵심 엔진입니다.
출처: Salesforce 공식 문서 Engagement: Journey Builder and Automation Studio (2026.06.23, 497p)

Automation Studio 개요

Automation Studio는 이메일 발송, 데이터 임포트, SQL 쿼리, 파일 전송 등 반복적인 마케팅·데이터 관리 작업을 자동화합니다. 단순한 단일 단계 자동화부터 복잡한 멀티스텝 워크플로까지 구성할 수 있습니다.

📌
Automation Studio vs Journey Builder: Automation Studio는 데이터 처리·배치 작업(SQL 쿼리, 파일 임포트, 발송 준비)에 사용하고, Journey Builder는 개인별 실시간 여정(이메일 시퀀스, Decision Split, Wait)에 사용합니다. 두 앱은 상호보완적으로 함께 사용됩니다.

시작 소스(Starting Source) 유형

유형트리거 조건대표 사용 케이스
Scheduled 지정된 날짜·시간·반복 주기 매일 새벽 2시 세그먼트 갱신, 매주 월요일 주간 발송 준비
File Drop Enhanced FTP의 Import/ 또는 Triggered_Automations/ 폴더에 파일 업로드 시 POS 데이터 야간 자동 임포트, 외부 시스템 파일 연동
Trigger API 외부 시스템(AWS S3, GCP, Azure)에 파일 업로드 → REST API 호출 실시간 트랜잭션 데이터 처리, 외부 이벤트 기반 자동화
Run Once 수동 즉시 실행 임시 세그먼트 생성, 긴급 발송 준비, 개별 Activity 테스트
💡
Advanced Run Once: Scheduled 자동화에서만 지원. 전체 자동화가 아닌 특정 스텝·액티비티만 선택해 실행할 수 있습니다. 에러가 발생한 액티비티만 수정 후 재실행할 때 유용합니다.

Activities 목록

자동화에서 사용할 수 있는 Activity 유형입니다. Activities 탭에서 미리 설정해두면 여러 자동화에서 재사용할 수 있습니다.

Send Email
Type ID: 42
이메일 발송. Send Definition 생성 후 다수 자동화에서 재사용 가능. STO, Delivery Window, Hourly Throttle 설정 지원
SQL Query
Type ID: 300
SQL로 Data View·DE를 쿼리해 결과를 대상 DE에 저장. Append / Update / Overwrite 3가지 Data Action 지원
Data Copy or Import
Type ID: 43
FTP 파일 또는 DE 데이터를 대상 DE로 임포트. 복잡한 SQL 없이 대용량 데이터 이동 가능
File Transfer
Type ID: 53
Enhanced FTP ↔ Safehouse 간 파일 이동, 압축 해제(unzip), 복호화(decrypt)
Data Extract
Type ID: 73
Tracking 데이터, DE 데이터를 CSV로 내보내기. FTP로 자동 전송 가능
Filter
Type ID: 303
Filter Activity로 구독자 필터링 후 대상 List/DE 갱신
Script
Type ID: 423
Server-Side JavaScript(SSJS) 실행. 복잡한 데이터 변환, REST API 호출 자동화
Refresh Group
Type ID: 45
Group(구독자 세그먼트)을 현재 데이터로 갱신
Publish Audience
Type ID: 427
Audience Builder의 Audience를 발행해 Sendable DE 업데이트
Wait
Type ID: 467
지정 시간만큼 자동화 일시 정지. 다음 단계 실행 전 대기
Verification
Type ID: 1000
DE 레코드 수 확인. 예상 범위 벗어나면 자동화 중단 또는 알림
Fire Event
Type ID: 749
Journey Builder의 API Event를 트리거해 여정에 Contact 진입
⚠️
1 Step = 1 Query 원칙: 하나의 Automation Step에는 Query Activity를 1개만 배치하세요. 같은 스텝에 여러 Query가 있으면 동일 데이터를 동시 접근(Concurrent Activity)해 타임아웃·실패가 발생할 수 있습니다.

SQL Query Activity

SQL Query Activity는 Data View와 DE를 SQL로 조회해 결과를 대상 DE에 저장하는 Automation Studio의 핵심 기능입니다. 세그먼트 갱신, 트래킹 데이터 수집, 재활성 대상 추출 등 다양한 데이터 파이프라인에 활용됩니다.

Query 작성 방법

1
Automation Studio → Activities → New Activity → SQL Query
2
쿼리 이름·External Key 입력 (External Key는 API 식별자로 활용)
3
SQL 작성: 왼쪽 패널에서 DE·Data View를 더블클릭 또는 드래그해 테이블·컬럼 추가 컬럼명 직접 입력 금지
4
Validate Syntax 클릭으로 문법 검증
5
대상 DE(Target Data Extension) 선택 — 쿼리에 사용한 DE와 달라야 함
6
Data Action 선택 (Append / Update / Overwrite) → 저장
7
자동화에 추가 후 스케줄 설정 및 활성화

Query 최적화 가이드

🚫
30분 타임아웃: SQL Query Activity는 30분 후 자동 실패합니다. 쿼리가 10분 이상 걸린다면 중간 테이블 분리 패턴을 적용하거나 Customer Data Platform 검토가 필요합니다.

성능 향상 핵심 원칙

  • JOIN + SELECT * 사용 금지: JOIN 시에는 반드시 컬럼명을 명시하세요. SELECT *는 단일 테이블에서만 허용
  • SARGable 쿼리 작성: WHERE절에 함수 감싸기 금지. WHERE YEAR(EventDate) = 2026 대신 WHERE EventDate >= '2026-01-01' 사용
  • LOWER() 금지: SQL Server는 대소문자를 구분하지 않으므로 LOWER() 함수는 불필요하며 성능을 저하시킴
  • Primary Key 남용 금지: 너무 많은 Primary Key는 INSERT 성능을 저하시킴. 조인 기준 필드만 지정
  • 정각 회피: 자동화 대부분이 정각에 집중됨. 8:00 → 8:17처럼 비정각으로 스케줄 분산
  • 독립 DE 사용: 자동화 전용 DE를 별도 생성해 다른 프로세스와 동시 접근 충돌 방지

Data Action 선택 기준

Data Action동작성능사용 적합 케이스
Overwrite 기존 데이터 전체 삭제 후 새 데이터 삽입 빠름 전체 세그먼트 갱신, 매일 새로 만드는 발송 대상 DE
Append 기존 데이터 유지, 새 데이터를 끝에 추가 보통 이벤트 로그 누적, 트래킹 이력 장기 보관
Update Primary Key 일치 시 업데이트, 없으면 추가 느림 회원 속성 최신값 유지, Upsert 패턴
💡
기본값은 Overwrite: 발송 대상 세그먼트처럼 매번 새로 만드는 DE에는 Overwrite가 가장 빠르고 안전합니다. Append는 오래 사용하면 DE 크기가 계속 커지므로 Retention Policy 설정을 꼭 함께 하세요.

Data Views

Data Views는 MCE가 자동 관리하는 시스템 테이블입니다. Automation Studio SQL Query Activity에서 [_ViewName] 형식으로 조회할 수 있습니다. 데이터는 최대 6개월 보관됩니다.

📌
시간대: 모든 Data View 날짜·시간은 Central Standard Time(CST) 기준입니다. Daylight Saving Time은 적용되지 않습니다. 한국 기준으로 보려면 DATEADD(hour, 15, EventDate)로 변환하세요.

주요 Data View 목록

_Sent
이메일 발송 이벤트. JobID, SubscriberKey, EventDate 포함
_Open
이메일 오픈 이벤트. IsUnique 필드로 중복 오픈 구분
_Click
이메일 링크 클릭 이벤트. LinkContent(AMPScript 변수 치환 후)·LinkName 포함
_Bounce
바운스 이벤트. BounceCategory, SMTPBounceReason 포함
_Unsubscribe
수신거부 이벤트. 스팸 신고도 포함. StatusChanges Extract로 전체 기간 조회 가능
_Complaint
스팸 신고(FBL) 이벤트. 발송 메시지와 JOIN 필요
_Subscribers
구독자 목록. Status(active/held/unsubscribed/bounced), BounceCount. Enterprise 레벨에서만 조회
_Job
이메일 발송 Job 정보. EmailName, EmailID, DeliveredTime. JobID 인덱스 필드
_Journey
Journey 버전 정보. JourneyName, VersionID, VersionNumber
_JourneyActivity
Journey 내 Activity 정보. ActivityName, ActivityType, JourneyActivityObjectID
_AutomationInstance
자동화 실행 이력. 성공/실패/Skip 상태, 에러 메시지. 자동화 헬스 모니터링에 활용
_BusinessUnitUnsubscribes
Enterprise 레벨 수신거부. Parent BU에서만 실행 가능
_ListSubscribers
List 구독자 목록. ListID 기준으로 구독자 조회
_FTAF
친구에게 전달(Forward to a Friend) 이벤트
_SMSSubscriptionLog
MobileConnect SMS 구독 이력. OptInStatus, OptInDate
📌
Enterprise 2.0 멀티 BU: 하위 BU에서 쿼리 시 Enterprise 레벨 Data View는 ent. 접두사를 추가하세요. 예: ent.[_Subscribers]. 없으면 해당 BU 데이터만 반환됩니다.

실전 SQL 패턴

바운스 이력 수집 (BounceHistory)

최근 3일 바운스 발생 구독자 목록을 수집해 별도 DE에 저장합니다. 매일 자동화로 실행해 24시간 이내 바운스를 CRM에 반영할 때 활용합니다.

SELECT
  b.SubscriberID,
  b.JobID,
  b.ListID,
  b.BatchID,
  b.EventDate,
  l.EmailAddress,
  b.SubscriberKey,
  j.EmailName,
  b.Domain,
  b.BounceCategory,
  b.SMTPBounceReason
FROM [_listsubscribers] AS l
INNER JOIN [_bounce] AS b
  ON l.SubscriberID = b.SubscriberID
INNER JOIN [_job] AS j
  ON b.JobId = j.JobID
WHERE b.EventDate > DATEADD(dd, -3, GETDATE())
💡
특정 이메일의 바운스만 추출하려면 마지막 줄에 AND j.EmailID = YOUR_EMAIL_ID를 추가하세요. Target DE Data Action: Update 권장.

최근 30일 고유 오픈 수집 (OpensIn30Days)

특정 이메일의 30일 이내 고유 오픈 목록을 추출합니다. 리인게이지먼트 대상 제외(최근 활성자 목록)나 세그먼트 갱신에 사용합니다.

SELECT
  j.JobID,
  j.EmailName,
  j.DeliveredTime AS SendTime,
  o.EventDate AS OpenTime,
  s.EmailAddress,
  s.SubscriberKey
FROM [_job] j
INNER JOIN [_open] o
  ON j.JobID = o.JobID
INNER JOIN [_subscribers] s
  ON o.SubscriberID = s.SubscriberID
WHERE o.IsUnique = 1
  AND o.EventDate > DATEADD(day, -30, GETDATE())
  AND j.EmailID = YOUR_EMAIL_ID

Journey 발송 일별 집계 (EmailSendsByJourney)

특정 Journey의 이메일 발송 수를 날짜·이메일명별로 집계합니다. 운영 리포팅 자동화에 활용합니다.

SELECT
  j.JourneyName,
  CAST(s.EventDate AS DATE) AS Date,
  ja.ActivityName AS EmailName,
  ja.ActivityExternalKey,
  COUNT(*) AS Sends
FROM [_sent] AS s
INNER JOIN [_journeyactivity] AS ja
  ON s.TriggererSendDefinitionObjectID
     = ja.JourneyActivityObjectID
INNER JOIN [_journey] AS j
  ON ja.VersionID = j.VersionID
INNER JOIN [_subscribers] AS su
  ON s.SubscriberID = su.SubscriberID
WHERE ja.ActivityType IN ('EMAIL', 'EMAILV2')
  AND j.JourneyName = 'YOUR_JOURNEY_NAME'
  AND s.EventDate > DATEADD(day, -7, GETDATE())
GROUP BY
  j.JourneyName, j.JourneyID,
  CAST(s.EventDate AS DATE),
  ja.ActivityName, ja.ActivityExternalKey
💡
오픈·클릭·바운스를 집계하려면 [_sent]를 [_open], [_click], [_bounce]로 교체하고, JOIN 조건의 s.를 해당 별칭으로 변경하면 됩니다. 모든 Journey 집계 시 마지막 AND j.JourneyName = ... 줄을 삭제하세요.

Journey 바운스 Contact 추출 (EmailBouncesByJourney)

Journey에서 이메일이 바운스된 Contact의 주소 정보를 추출해 다이렉트 메일 발송에 활용하는 패턴입니다.

SELECT
  j.JourneyName,
  j.VersionNumber,
  ja.ActivityName AS EmailName,
  s.EventDate   AS SendTime,
  b.EventDate   AS BounceTime,
  su.EmailAddress,
  su.SubscriberKey AS ContactKey,
  cpd.AddressLine1,
  cpd.City, cpd.State, cpd.Zip
FROM [_sent] AS s
INNER JOIN [_journeyactivity] AS ja
  ON s.TriggererSendDefinitionObjectID
     = ja.JourneyActivityObjectID
INNER JOIN [_journey] AS j
  ON ja.VersionID = j.VersionID
INNER JOIN [_bounce] AS b
  ON  s.JobID        = b.JobID
  AND s.ListID       = b.ListID
  AND s.BatchID      = b.BatchID
  AND s.SubscriberID = b.SubscriberID
INNER JOIN [_subscribers] AS su
  ON b.SubscriberID = su.SubscriberID
INNER JOIN [ContactProfileData] AS cpd
  ON su.SubscriberKey = cpd.ContactKey
WHERE ja.ActivityType IN ('EMAIL', 'EMAILV2')
  AND j.JourneyName = 'YOUR_JOURNEY_NAME'
📌
ContactProfileData는 사용자 정의 DE로, 주소 필드가 있는 회원 정보 DE를 JOIN합니다. 실제 환경의 DE명으로 교체하세요.

중간 테이블 패턴 (장기 쿼리 분리)

여러 Data View를 한 번에 JOIN하면 30분 타임아웃 위험이 있습니다. 단계별 중간 테이블로 분리하면 각 쿼리 실행 시간이 단축됩니다.

StepQuery대상 DE설명
Step 1JobTable QueryJobTable_job에서 특정 이메일 Job 정보 추출
Step 2OpenTable QueryOpenTable_open에서 고유 오픈 JobID·SubscriberID 추출
Step 3SubscriberTable QuerySubscriberTableOpenTable과 _subscribers를 JOIN해 이메일 주소 포함
Step 4Final QueryOpensIn30Days위 3개 중간 DE를 JOIN해 최종 결과 생성. Data Action: Overwrite
💡
각 Step은 독립 DE에 Overwrite로 저장하고, 다음 Step에서 그 DE를 참조합니다. Step 1→2→3→4 순서가 보장되므로 하나의 자동화 내 순차 Step으로 구성하세요.

Journey Builder 개요

Journey Builder는 Contact의 행동·속성·시간에 따라 자동으로 분기하고 개인화 메시지를 발송하는 고객 여정 자동화 도구입니다. Single Send, Transactional Send, Multi-Step Journey 세 유형을 지원합니다.

⚠️
사전 조건 확인: Journey Builder에서 Contact 속성(회원등급, 구매일 등)을 Decision Split 조건으로 사용하려면 해당 DE가 반드시 Contact Builder Attribute Group에 연결되어 있어야 합니다. 미설정 시 속성값이 빈 값으로 반환됩니다.

Entry Sources — 진입 소스

Entry Source는 어떤 Contact가 Journey에 진입할지 결정합니다. 캔버스 상단에 반드시 1개 배치해야 합니다.

Entry Source트리거대표 케이스
Data Extension 지정 DE의 Contact. 배치(스케줄) 또는 수동 진입 주간 뉴스레터, 세그먼트 캠페인, 생일 여정(매일 스케줄)
API Event 외부 시스템이 REST API 호출 시 즉시 진입 회원가입 환영, 구매 확인, 장바구니 이탈(실시간)
Salesforce Data Event CRM의 Lead/Contact/Opportunity 레코드 생성·변경 시 Lead 생성 → 환영 이메일, Opportunity Stage 변경 → 알림
Salesforce Audience Audience Builder 세그먼트 갱신 시 변경된 Contact 등급 변경 고객, VIP 승급 알림
CloudPages CloudPages Form 제출 시 이벤트 등록 후속 여정, 설문 응답 분기
Inbound Chat WhatsApp·SMS 수신 시 키워드 응답 기반 자동화

Data Extension Entry Source 설정

1
Canvas에 Data Extension 드래그
2
Sendable DE 선택 (Contact Key가 있는 DE)
3
Filter Contacts: 추가 조건으로 진입 범위 좁히기 (옵션)
4
Schedule 설정: 1회 / 반복 / 수동. 반복 선택 시 Re-entry 기간 설정
5
Summary 검토 → Done
📌
Entry Source DE 설계 원칙: 여러 DE 데이터가 필요하면 Automation Studio SQL Query로 먼저 단일 DE에 병합하고 그 DE를 Entry Source로 사용하세요. Filter 조건에 사용할 속성도 Entry Source DE에 포함되어야 합니다 — Contact Builder 속성보다 Entry DE 속성을 우선 활용하면 처리 속도가 빠릅니다.

API Event Entry Source

외부 시스템(웹사이트, 앱, CRM)에서 REST API를 호출해 즉시 Journey에 Contact를 진입시킵니다. 실시간 트리거가 필요한 여정(가입 환영, 구매 완료)에 필수입니다.

1
Journey Builder → Create Event → API Event 생성 (이름·External Key 설정)
2
Event에 포함할 데이터 필드 정의 (이름, 이메일, 주문번호 등)
3
Canvas에 API Event 드래그 → 생성한 Event 선택
4
Journey 활성화 후 외부 시스템에서 REST API 호출:
POST /interaction/v1/events → EventDefinitionKey + ContactKey + 데이터 페이로드
💡
Automation Studio의 Fire Event Activity(Type ID: 749)를 사용하면 배치 DE에서 대량 Contact를 API Event Journey에 진입시킬 수 있습니다. 실시간 API와 배치 자동화를 함께 사용할 때 활용합니다.

Salesforce Data Event

CRM의 Lead/Contact 레코드 생성·필드 변경 시 자동으로 Journey에 진입합니다. Marketing Cloud Connect 필요.

⚠️
APEX 한도 주의: Salesforce Data Event는 Flow Builder/Process Builder와 함께 동일 Object Save 순서에 실행됩니다. APEX Future 호출 50건 한도, DML 한도 등 APEX Governor Limits 초과 위험이 있습니다. 복잡한 CRM 자동화가 이미 있는 org에서는 설계 전 검토가 필요합니다.

Split Activities — 분기

🔀
Engagement Split

이전 이메일의 오픈·클릭 여부로 분기. "이메일 열람한 사람 → A 경로, 미열람 → B 경로". 이메일 Activity 다음에만 배치 가능

📊
Decision Split

Contact 속성(등급, 구매일, 지역 등) 조건으로 최대 10개 경로 분기. Entry Source DE 또는 Contact Builder Attribute Group 필드 사용. 가장 많이 쓰이는 분기 유형

🎲
Random Split

Contact를 지정 비율로 무작위 분배. A/B 테스트, 대조군 설정, 트래픽 분산에 사용. 최대 10개 경로, 합계 100%

🏆
Path Optimizer

여러 이메일 버전 중 성과 기준(오픈율·클릭율)으로 자동 선정. 일정 기간 후 승자 경로에 나머지 Contact 자동 발송

Wait Activities — 대기

⏳
Wait by Duration

지정 일수·시간 대기 후 다음 단계 진행. "3일 후", "1주일 후" 패턴의 기본 Wait

📅
Wait by Attribute

Contact의 날짜 속성(생일, 만료일)을 기준으로 대기. "생일 3일 전", "만료일 당일" 시점에 다음 단계 실행

🕐
Wait Until

특정 날짜·시간까지 대기 후 진행. 이벤트 당일, 시즌 오픈일 등 고정 날짜 기준 여정에 활용

성능 최적화

📌
HTS(High Throughput Sending): 이메일 Activity가 있는 모든 Journey에 HTS를 활성화하세요. HTS는 처리량을 높이고 병렬 Journey 실행 시 지연을 줄입니다. HTS 미활성 Journey가 하나라도 있으면 전체 계정 성능에 영향이 생길 수 있습니다.
  • Entry Source 필터 최소화: Entry Source에 Filter를 추가하면 진입 처리 속도가 낮아집니다. 가능하면 Automation Studio Query에서 미리 필터링된 DE를 사용하세요
  • Decision Split → Contact 속성 우선: Entry Source DE에 속성이 포함되어 있으면 Contact Builder 조회 없이 바로 분기. 속성을 Entry DE에 포함시키세요
  • Journey 시작 시간 분산: 여러 Journey가 동시에 시작하면 처리 자원 경쟁이 발생합니다. System Priority 설정과 시작 시간 스태거링으로 완화하세요
  • One-to-Many DE 주의: Contact Builder에 One-to-Many 관계 DE가 연결되어 있으면 Decision Split에서 같은 속성값이 여러 번 나타날 수 있습니다
  • 30일 이상 쿼리 타임아웃: Entry Source DE를 만들 때 30일 이상 범위 쿼리는 타임아웃 위험이 있습니다. 중간 테이블 패턴을 사용하세요

Unsubscribe 처리
-- 90일 이상 비활성 구독자 추출
-- 최근 90일 내 오픈 또는 클릭이 없는 구독자
SELECT DISTINCT
  s.SubscriberKey,
  s.EmailAddress
FROM [MasterDE] s
WHERE s.SubscriberKey NOT IN (
  SELECT SubscriberKey
  FROM [_open]
  WHERE EventDate > DATEADD(day, -90, GETDATE())
    AND IsUnique = 1
)
AND s.SubscriberKey NOT IN (
  SELECT SubscriberKey
  FROM [_click]
  WHERE EventDate > DATEADD(day, -90, GETDATE())
    AND IsUnique = 1
)
⚠️
오픈 데이터 신뢰도 주의: Apple Mail Privacy Protection 등으로 인해 오픈 데이터는 과대 집계될 수 있습니다. 재활성화 여부는 오픈보다 클릭을 기준으로 판단하는 것이 더 정확합니다. Engagement Split 대신 Decision Split으로 _click 기반 클릭 여부를 확인하세요.
